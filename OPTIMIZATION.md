# MassOVA 功能优化建议

## 高优先级

### 1. 任务持久化

**现状：** 任务存储在内存 `Map` 中（`server/jobs.js:5`），服务重启后所有任务记录丢失。

**建议：** 将任务数据写入 SQLite 或 JSON 文件，重启后可恢复历史任务状态和日志。

---

### 2. 失败重试 / 断点续部署

**现状：** 批次中某台 VM 部署失败后，只能取消整批任务（`server/jobs.js:89`），无法单独重试失败项。

**建议：** 任务完成后增加 "重试失败项" 按钮，仅对 `failed` 状态的 VM 重新执行部署，避免全部重来。

---

### 3. 并发部署

**现状：** 严格串行逐台部署（`server/jobs.js:65` 的 `for...of` 循环），大批量场景耗时长。

**建议：** 增加并发数配置项（如 3-5 并行），使用 `Promise.all` 配合并发池控制同时运行的 ovftool 进程数量，显著提升大批量部署速度。

---

### 4. 实时日志推送

**现状：** 前端每 1.8 秒轮询 `/api/jobs` 拉取任务状态和日志（`src/main.jsx:89`）。

**建议：** 改用 SSE（Server-Sent Events）或 WebSocket 实现日志流式推送，减少不必要的网络开销，日志输出更实时。

---

### 5. 部署配置模板

**现状：** 用户每次部署都需要重新填写连接信息、网络映射、命名规则等全部参数。

**建议：** 增加 "保存为模板" 功能，将完整的连接配置、网络映射、VM 命名规则持久化存储，下次部署时一键加载。模板可存在 localStorage 或后端数据库。

---

## 中优先级

### 6. Web 界面认证

**现状：** 无任何访问控制，任何能访问服务端口的人都可以执行 VM 部署操作。

**建议：** 增加简单的 Basic Auth 或 Token 认证机制，防止未授权访问。

---

### 7. VM 名称冲突检测

**现状：** 提交部署前不检查目标路径下是否已存在同名 VM，依赖 ovftool 运行时报错。

**建议：** 在提交前通过 vSphere API 查询目标 folder 下已有的 VM 列表，与即将部署的 VM 名称做比对，提前在 UI 层给出冲突警告。

---

### 8. Datastore 容量校验

**现状：** 连接读取资源时不获取 datastore 的可用空间信息。

**建议：** 在 discover 阶段额外获取 datastore 的 `freeSpace` 属性，结合模板磁盘大小和部署数量估算所需空间，空间不足时提前预警。

---

### 9. 每台 VM 独立配置

**现状：** 所有 VM 共享同一套网络映射和 OVF 属性，无法为单台 VM 定制 IP、hostname 等。

**建议：** 在手动清单模式下，支持 CSV 列导入（如 `name,ip,gateway`），允许为每台 VM 设置不同的 OVF 属性值。生成的 ovftool 命令中按 VM 传入对应的 `--prop` 参数。

---

### 10. 日志导出

**现状：** 任务日志只能在前端页面查看，无法导出。

**建议：** 增加 "导出日志" 按钮，将任务的完整日志（命令、输出、错误信息）导出为文本文件，方便审计和排障。

---

## 低优先级 / 体验优化

### 11. 可视化进度条

**现状：** 进度以文字 `completed/total` 展示（`src/main.jsx:547`）。

**建议：** 增加可视化进度条组件，直观展示当前批次部署进度。

---

### 12. vSphere 会话复用

**现状：** 每次 discover 都重新执行 `RetrieveServiceContent` + `Login`（`server/vsphere.js:6-9`），短时间内多次操作会反复建立连接。

**建议：** 对同一 host+username 组合缓存 session cookie，设置合理的 TTL（如 10 分钟），减少重复登录开销。

---

### 13. 部署后自定义脚本

**现状：** 部署完成后仅支持 `--powerOn` 选项（`server/ovftool.js:67`）。

**建议：** 支持配置部署后自动执行的 guest customization 或自定义脚本，如通过 VMware Tools 执行 guest 内命令。

---

### 14. 多模板源

**现状：** 一次部署任务只能选择一个 vSphere 模板作为源（`server/index.js:120`）。

**建议：** 支持在同一个批次中混合不同模板，每台 VM 可指定不同的模板源，满足异构批量部署需求。

---

### 15. 国际化 (i18n)

**现状：** UI 文案全部为中文硬编码在 JSX 中。

**建议：** 引入 i18n 框架（如 react-intl 或 i18next），将文案抽为资源文件，支持多语言切换。
