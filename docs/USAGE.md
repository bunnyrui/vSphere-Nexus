# 使用文档

## 概述

MassOVA 是一个从 vSphere 模板批量部署虚拟机的 Web 工具。它通过 `ovftool` 命令行工具与 ESXi/vCenter 交互，支持并发部署、实时日志、失败重试等功能。

### 核心限制

- **部署源必须是 vSphere 模板**，不支持本地上传 OVA 文件
- 如果只有 OVA 文件，需要先手动导入到 vSphere 并转换为模板

---

## 快速开始

1. 打开 MassOVA Web 界面（默认 http://localhost:4173）
2. 如果启用了认证，先输入用户名和密码登录
3. 填写 vSphere 连接信息
4. 点击 "连接并读取资源"
5. 选择模板、配置参数
6. 生成 VM 清单
7. 点击 "生成部署预览"（干跑模式）或 "开始批量部署"

## 操作步骤详解

### 第一步：连接 vSphere

1. **选择平台类型**：ESXi（单主机直连）或 vCenter（集中管理）
2. **填写连接信息**：
   - vSphere 地址：ESXi 或 vCenter 的 IP/域名
   - 用户名：vSphere 登录用户名（ESXi 默认 `root`）
   - 密码：vSphere 登录密码
3. **点击 "连接并读取资源"**

连接成功后，系统会自动读取以下资源并填充为下拉选项：

- 部署目标（vCenter 模式）/ 主机信息
- Datastore（显示可用空间）
- VM Folder
- 可用网络
- 可用模板

### 第二步：配置部署参数

#### 连接与目标

| 参数 | 说明 |
|------|------|
| 平台类型 | ESXi 或 vCenter |
| vSphere 地址 | 目标主机 IP 或域名 |
| Datastore | 存储，下拉选择，显示可用空间 |
| VM Folder | 虚拟机存放的文件夹（可选） |
| 磁盘模式 | `thin`（精简）、`thick`（厚置备）、`eagerZeroedThick`（厚置备快速归零） |
| 部署后开机 | 勾选后部署完成自动开机 |

#### 部署源

从下拉列表中选择一个 vSphere 模板。连接后只显示模板类型的虚拟机，不显示普通 VM。

#### 网络映射

将模板中的源网络映射到目标端口组：

| 列 | 说明 |
|----|------|
| 模板源网络 | 从模板中自动读取的网络适配器 |
| ESXi Port Group | 目标环境中的端口组 |

选择模板后，源网络会自动填充。点击 "+" 添加更多映射。

#### OVF 属性

如果模板包含 OVF 属性（如 IP 地址、主机名等），可以在此设置值。支持模板变量：

- `{{index}}` — 替换为从 1 开始的序号
- `{{zeroIndex}}` — 替换为从 0 开始的序号
- `{{name}}` — 替换为 VM 名称

#### 并发数

控制同时运行的 ovftool 进程数量（1-10）。建议值：

- 本地网络 / SSD 存储：3-5
- 低带宽 / 机械硬盘：1-2
- 高性能存储 / 万兆网络：5-10

### 第三步：生成虚拟机清单

#### 简单生成模式

适合规则命名的批量部署：

| 参数 | 示例 | 生成的名称 |
|------|------|------------|
| 名称前缀 | `web` | - |
| 数量 | `3` | - |
| 起始编号 | `1` | - |
| 编号位数 | `2` | `web-01`、`web-02`、`web-03` |

更多示例：

| 前缀 | 数量 | 起始 | 位数 | 结果 |
|------|------|------|------|------|
| `db` | 5 | 1 | 3 | `db-001` ~ `db-005` |
| `node` | 10 | 0 | 2 | `node-00` ~ `node-09` |
| `test` | 1 | 1 | 1 | `test-1` |

#### 手动清单模式

适合不规则命名或需要精确控制每台 VM 名称的场景：

- 在文本框中一行输入一个名称
- 支持 `{{index}}` 变量，提交时自动替换为序号
- 点击 "新增一台" 快速添加一行

### 第四步：部署

#### 干跑模式（默认开启）

点击 **"生成部署预览"**，系统会：

1. 生成所有 VM 的 ovftool 命令（密码脱敏显示）
2. 记录到任务面板
3. **不会实际执行部署**

建议每次正式部署前先干跑一次，确认命令和参数正确。

#### 提交前检查

提交时系统会自动检查：

1. **VM 名称冲突**：如果目标环境中已有同名 VM，会弹出警告
   - 点击 "返回修改" 回去改名称
   - 点击 "仍然部署" 强制提交（ovftool 可能会覆盖或报错）
2. **Datastore 容量**：根据模板大小和部署数量估算所需空间，空间不足时显示警告

#### 正式部署

1. 关闭 "干跑模式" 开关
2. 点击 **"开始批量部署"**
3. 任务面板实时显示进度和日志

## 任务管理

### 查看任务

左侧边栏显示所有历史任务列表，点击切换查看。任务面板显示：

- 任务状态和进度条
- ovftool 命令预览（最多显示前 5 条）
- 实时日志流
- 导出按钮

### 任务状态

| 状态 | 含义 |
|------|------|
| `queued` | 排队中 |
| `running` | 正在部署 |
| `succeeded` | 全部成功 |
| `failed` | 部分或全部失败 |
| `cancelled` | 用户取消 |
| `interrupted` | 服务重启导致中断 |

### 取消任务

任务运行中时，任务面板右上角出现取消按钮（X 图标）。点击后立即中止所有正在运行的 ovftool 进程。

### 重试失败项

任务状态为 `failed` 时，任务面板出现重试按钮（循环箭头图标）。点击后仅重新部署失败的 VM，成功的不会重复部署。

### 导出日志

点击导出按钮（下载图标），将任务完整日志导出为 `.log` 文本文件。

## 配置模板

### 保存模板

1. 配置好所有参数（连接信息、模板、网络映射、命名规则等）
2. 点击顶部 "保存模板" 按钮
3. 输入模板名称，点击保存

模板保存在服务端（`data/templates.json`），不包含 vSphere 密码。

### 加载模板

点击顶部 "加载模板..." 下拉框，选择已保存的模板。所有参数会自动填充（密码需要重新输入）。

### 删除模板

目前通过 API 删除：

```bash
curl -X DELETE http://localhost:4173/api/templates/模板名称
```

## vCenter 目标路径

vCenter 模式下，连接后会自动读取 inventory 并生成部署目标路径。常见路径格式：

```
/Datacenter/host/172.16.109.3/Resources              # 资源池（最常用）
/Datacenter/host/ClusterName/Resources                # 集群资源池
/Datacenter/host/172.16.109.3/Resources/子资源池名称    # 子资源池
```

ESXi 直连模式下不需要目标路径，系统自动处理。

## 认证登录

如果服务端配置了 `MASSOVA_USER` 和 `MASSOVA_PASS` 环境变量，访问 Web 界面时会先显示登录页面。

- 输入正确的用户名和密码后获取 token
- token 存储在浏览器 localStorage 中
- 关闭浏览器后需要重新登录
- 如果长时间不操作导致 token 失效，刷新页面会自动跳转到登录页

## 数据安全

- vSphere 密码仅保存在浏览器 sessionStorage 中（关闭标签页即清除）
- 任务日志中的 ovftool 命令密码已脱敏（只显示首字符 + 星号）
- `data/payloads.json` 包含原始密码用于重试功能，注意文件访问权限
- 通信建议在生产环境使用 HTTPS（通过 Nginx 反向代理配置 SSL）

## 常见问题

### 部署失败，日志显示 "Already exists"

目标环境中已有同名 VM。解决方式：

1. 在 vSphere Client 中删除或重命名冲突的 VM
2. 修改 MassOVA 中的 VM 名称

### 连接成功但看不到模板

确认目标环境中确实存在模板类型的虚拟机。普通 VM 不会出现在模板列表中。在 vSphere Client 中右键虚拟机 → "转换为模板"。

### 网络映射中没有源网络

模板可能没有配置网络适配器。在 vSphere Client 中编辑模板设置，添加网络适配器。

### 部署速度慢

1. 调高并发数（建议 3-5）
2. 确认网络带宽是否充足
3. 确认 Datastore 的 IO 性能
4. 使用 `thin` 磁盘模式比 `thick` 更快

### 重启服务后任务变成 "interrupted"

这是正常行为。服务重启时，正在运行的任务无法恢复执行。可以：

1. 查看任务日志确认哪些 VM 已经部署成功
2. 使用重试功能重新部署失败的 VM
